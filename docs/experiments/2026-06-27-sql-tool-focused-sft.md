## Hypothesis

For data-agent use, the next SFT should focus on:

1. Exact `execute_sql` tool-call arguments.
2. SQL schema linking and executable SQL style.
3. MCP/tool routing preservation.
4. Reduced influence from older mixed/weak-format data.

## 2026-06-28 Phase7 GRPO parallelization notes

Goal: replace the initial single-PPU WikiSQL execution-reward GRPO smoke run with a faster 16-PPU setup.

What was attempted:

1. Single-PPU TRL GRPO smoke
   - Script: `scripts/train_grpo_wikisql_exec.py`
   - Base: Phase5 merged Qwen2.5-Coder-7B LoRA SFT model.
   - Reward: SQL extraction, SQLite execution success, execution-result exact match, normalized SQL exact match.
   - Status: training was healthy but slow, about several seconds per rollout step, because rollout generation is on one PPU.

2. TRL/Accelerate 16-process DDP
   - Script: `scripts/train_grpo_wikisql_exec_accel16.py`
   - Launcher: `/opt/ac2/bin/accelerate launch --num_processes 16 --multi_gpu --mixed_precision bf16 ...`
   - Result: model loaded and entered the first reward step, then failed in PPU/NCCL all-reduce with `ncclUnhandledCudaError` / `driver shutting down`.
   - Interpretation: current PPU + TRL/Accelerate GRPOTrainer path is not stable for 16-rank GRPO on this job.

3. Independent per-PPU shard parallelism
   - Plan: run 16 independent single-PPU GRPO shards with disjoint WikiSQL offsets and compare adapters after evaluation.
   - Result: a single 64-sample health shard on GPU0 completed successfully; however launching many independent Python processes across different PPUs caused `cudaGetDeviceCount` / `driver shutting down` errors.
   - Interpretation: this PPU runtime appears to prefer a single managed multi-rank job instead of many unrelated Python processes with separate `CUDA_VISIBLE_DEVICES` bindings.

Current conclusion:

- The bottleneck diagnosis is correct: GRPO is rollout-bound and single PPU is too slow for full experiments.
- For this job, the stable path should reuse the already validated `/opt/ac2/bin/torchrun` 16-rank entry used by Phase5/Phase6 SFT, not Accelerate and not multiple independent shell jobs.
- Next implementation target: a custom WikiSQL execution-reward LoRA GRPO script based on `scripts/remote/train_lora_sft.py` + `scripts/remote/train_verifiable_grpo.py`, using DDP under torchrun, LoRA-only optimizer, per-rank rollout, rank0 metrics, and SwanLab logging.
- If torchrun GRPO also fails in NCCL, fall back to single-PPU GRPO only for smoke/evidence collection and switch the real RL stage to verl/SGLang or ms-swift after validating the PPU-supported launcher.
